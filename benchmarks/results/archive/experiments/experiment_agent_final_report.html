Key Findings

What this experiment reveals about agent-based search

Q13: 5 vs 2

Multi-search is GRAIL's advantage

When the answer spans multiple legal provisions (accreditation + convention + D54 process), GRAIL's ability to make two targeted cascade searches with refined queries finds details that a single RAG search misses. The agent intelligently iterates — first for general requirements, then specifically for the approval process.

Q09: 2 vs 5

Wrong tool choice hurts

GRAIL chose local_search with an entity-description query for Q09, but the specific criteria (epidemiological studies, cost-effectiveness) are in raw article text, not entity descriptions. Cascade would have found them via BM25 text matching. The agent needs better heuristics for when details are in text vs entities.

Context volume matters

GRAIL agent gets 10-12K tokens of structured context per search (entities + relationships + communities + text units). RAG agent gets 4K tokens of raw chunks. When the answer requires connecting multiple concepts, the richer context helps. When it's a single fact in one chunk, both suffice.

Tool selection patterns

GRAIL: cascade ×7, local ×1. The agent follows the "factual → cascade" heuristic. The one local_search (Q09) was a reasonable choice but suboptimal for this specific question.
RAG: rag_search ×5. Always the same tool — no strategic decisions.

Next Steps

Improvements identified from this experiment

1. Fix Q09 pattern

When the agent picks local_search but gets an incomplete answer, it should retry with cascade_search on the second call. Currently it doesn't iterate when local results seem sufficient but miss key details.

2. Fix Q10 pattern

Both agents fail to surface specific numbers (cien mil millones) despite having them in context. The system prompt should instruct the agent to always include specific amounts, deadlines, and article numbers from the retrieved context.
