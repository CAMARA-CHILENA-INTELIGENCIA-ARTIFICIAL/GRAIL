Por Qué Importa Este Benchmark

El cambio de precisión de recuperación a profundidad de comprensión

En la era agéntica, el problema de recuperación de RAG está mayormente resuelto. Los modelos de embeddings modernos encuentran chunks relevantes de forma confiable. El verdadero desafío es lo que pasa después de la recuperación: ¿puede el sistema comprender las relaciones entre conceptos, conectar información entre documentos y sintetizar respuestas que requieren comprensión — no solo encontrar — el material fuente?

Este benchmark evalúa exactamente eso. Un paciente con cáncer que pregunta "¿de dónde sale el dinero para mi tratamiento?" no necesita que el sistema encuentre el Artículo 20 — necesita que entienda que el Fondo tiene 4 fuentes de financiamiento, cómo se conectan con FONASA y qué protecciones existen. Eso requiere cruzar entidades, seguir cadenas de relaciones y sintetizar desde conocimiento estructurado — no solo devolver el chunk de texto más cercano.

La ventaja de GRAIL viene de su infraestructura de comprensión: entidades con descripciones semánticas, relaciones que codifican cómo se conectan los conceptos, reportes de comunidad que pre-resumen temas y consultas de recuperación que conectan el lenguaje del paciente con el conocimiento experto. El agente luego elige estratégicamente la herramienta correcta para cada tipo de pregunta — algo que un agente RAG de una sola herramienta no puede hacer.

Categorías del Benchmark

7 categorías para aislar capacidades de recuperación

Hecho Puntual
5 preguntas · fácil-medio
+0.52
Qué evalúa: Can the system locate one specific fact buried in one chunk? This is the baseline — if GRAIL can't match RAG here, the graph layer is overhead.
Por qué gana GRAIL: Entity descriptions with retrieval queries match patient language better than raw legal text. "¿Qué es la Ley Ricarte Soto?" matches the entity description directly.
Ejemplo: "¿Cada cuánto tiempo se revisa qué tratamientos se consideran de alto costo?"
Multi-Chunk
5 preguntas · medio-difícil
+0.72
Qué evalúa: The answer is scattered across 2+ chunks of the same document. RAG retrieves top-10 chunks by similarity but may miss the second relevant passage if it's far from the first.
Por qué gana GRAIL: Entity connections pre-link related chunks. When the agent finds "FONDO DE ALTO COSTO", the entity's text_unit_ids point to ALL chunks that mention it — including the one with funding sources (Art. 20).
Ejemplo: "¿De dónde sale el dinero para pagar los tratamientos caros?"
Cruce de Fuentes
5 preguntas · hard
+0.80
Qué evalúa: The answer requires combining information from 2+ different documents. RAG's cosine similarity over chunks rarely bridges across documents — it finds the best chunks from one file.
Por qué gana GRAIL: The entity graph connects "FONASA" across all 3 laws. When the agent searches for provider requirements, entities link Ley 20.850 (legal basis) → Decreto 54 (operational details) automatically.
Ejemplo: "¿Qué necesita un hospital para atender pacientes bajo la Ley Ricarte Soto?"
Procedimental
4 preguntas · medio-difícil
+0.64
Qué evalúa: "What are the steps to…?" — ordered sequences that span multiple articles. RAG may return the right chunks but in the wrong order, losing the procedural flow.
Por qué gana GRAIL: Relationship chains encode process flow: Subsecretaría → evaluación → Comisión → recomendación → decreto. The agent uses local+cascade to capture both the process entity and the specific steps.
Ejemplo: "Si un hospital me hizo daño, ¿qué tengo que hacer antes de poder demandarlo?"
Comparativa
3 preguntas · fácil-difícil
+0.80
Qué evalúa: "What's the difference between X and Y?" where X and Y are defined in different places. A single RAG search finds one side of the comparison; the other side requires a second, well-targeted search.
Por qué gana GRAIL: The agent makes two local_search calls — one per concept — each crafted with the WHO+WHAT+TERMS formula. RAG searches once and hopes both sides land in the top-10 chunks.
Ejemplo: "¿Quiénes asesoran sobre garantías de salud y quiénes vigilan el sistema de alto costo? ¿Son los mismos?"
Negación / Límite
3 preguntas · fácil-medio
+0.08
Qué evalúa: "Does the law cover X?" where the answer is NO. Tests whether the system retrieves exclusion clauses. Embedding similarity may miss negations because the query is semantically close to the covered concept.
Casi empatado: Both systems handle explicit exclusions well. The exclusion clauses (Art. 1 "se excluyen") are prominent in the source text and match well with both entity descriptions and chunk similarity.
Ejemplo: "¿La Ley Ricarte Soto cubre los accidentes laborales o de tránsito?"
Síntesis Global
5 preguntas · fácil-difícil
+0.90
Qué evalúa: Bird's-eye view of the entire corpus. "What is this about?", "What rights do I have?", "How do these 3 laws connect?" RAG has no single chunk that answers these — the answer IS the corpus.
Por qué gana GRAIL (mayor delta): Global search reads pre-computed community report summaries that cover the entire knowledge base thematically. It answers "¿de qué trata?" in 6 seconds. RAG must piece together an overview from random top-10 chunks.
Ejemplo: "Haz un resumen de cómo estas tres normativas protegen a un paciente de cáncer."

Replica Este Benchmark

Ejecuta con el dataset incluido o trae tu propio corpus

Con el dataset incluido

# 1. Indexa el corpus de leyes oncológicas
grail index benchmark_laws

# 2. Ejecuta el benchmark (solo agentes)
python benchmarks/run_benchmark.py \
  --config benchmark_laws/grail.yaml \
  --benchmark benchmarks/simple_benchmark/benchmark.json \
  --skip-rag --include-agents --skip-judge

# 3. Resultados guardados en benchmarks/results/<timestamp>/

Con tu propio corpus

# 1. Crea e indexa tu proyecto
grail init my_project
cp my_documents/* my_project/input/
grail index my_project

# 2. Escribe un benchmark.json con tus preguntas
#    (ver benchmarks/simple_benchmark/ para formato)

# 3. Ejecutar
python benchmarks/run_benchmark.py \
  --config my_project/grail.yaml \
  --benchmark my_benchmark.json \
  --skip-rag --include-agents --skip-judge

Opciones de Evaluación

LLM como Juez (automatizado)
python benchmarks/run_benchmark.py \
  --config benchmark_laws/grail.yaml \
  --benchmark benchmarks/simple_benchmark/benchmark.json \
  --include-agents \
  --judge-model "openai|gpt-4o"

Cualquier endpoint compatible con OpenAI funciona. El juez evalúa cada respuesta en 5 dimensiones contra respuestas gold. Rápido, económico, reproducible.

Claude Code como Juez (manual, máxima calidad)
# 1. Ejecutar with --skip-judge
python benchmarks/run_benchmark.py \
  --config benchmark_laws/grail.yaml \
  --benchmark benchmarks/simple_benchmark/benchmark.json \
  --include-agents --skip-judge

# 2. Pide a Claude Code: "evalúa el benchmark"
# Claude lee responses.json + benchmark.json,
# evalúa cada respuesta, escribe judge_scores.json

Evaluación de máxima calidad — Claude tiene contexto completo de los documentos fuente y entiende matices del dominio. Usado para este benchmark.
