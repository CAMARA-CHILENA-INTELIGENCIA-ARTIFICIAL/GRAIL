Why This Benchmark Matters

The shift from retrieval accuracy to comprehension depth

In the agentic era, RAG's retrieval problem is largely solved. Modern embedding models find relevant chunks reliably. The real challenge is what happens after retrieval: can the system comprehend relationships between concepts, connect information across documents, and synthesize answers that require understanding — not just finding — the source material?

This benchmark tests exactly that. A cancer patient asking "¿de dónde sale el dinero para mi tratamiento?" doesn't need the system to find Article 20 — they need it to understand that the Fund has 4 funding sources, how they connect to FONASA, and what protections exist. That requires cross-referencing entities, following relationship chains, and synthesizing from structured knowledge — not just returning the closest text chunk.

GRAIL's advantage comes from comprehension infrastructure: entities with semantic descriptions, relationships that encode how concepts connect, community reports that pre-summarize themes, and retrieval queries that bridge patient language to expert knowledge. The agent then strategically picks the right tool for each question type — something a single-tool RAG agent cannot do.

Benchmark Categories

7 categories designed to isolate different retrieval capabilities — from simple lookups to full-corpus synthesis

Single Fact
5 questions · easy-medium
+0.52
What it tests: Can the system locate one specific fact buried in one chunk? This is the baseline — if GRAIL can't match RAG here, the graph layer is overhead.
Why GRAIL wins: Entity descriptions with retrieval queries match patient language better than raw legal text. "¿Qué es la Ley Ricarte Soto?" matches the entity description directly.
Example: "¿Cada cuánto tiempo se revisa qué tratamientos se consideran de alto costo?"
Multi-Chunk
5 questions · medium-hard
+0.72
What it tests: The answer is scattered across 2+ chunks of the same document. RAG retrieves top-10 chunks by similarity but may miss the second relevant passage if it's far from the first.
Why GRAIL wins: Entity connections pre-link related chunks. When the agent finds "FONDO DE ALTO COSTO", the entity's text_unit_ids point to ALL chunks that mention it — including the one with funding sources (Art. 20).
Example: "¿De dónde sale el dinero para pagar los tratamientos caros?"
Cross-Source
5 questions · hard
+0.80
What it tests: The answer requires combining information from 2+ different documents. RAG's cosine similarity over chunks rarely bridges across documents — it finds the best chunks from one file.
Why GRAIL wins: The entity graph connects "FONASA" across all 3 laws. When the agent searches for provider requirements, entities link Ley 20.850 (legal basis) → Decreto 54 (operational details) automatically.
Example: "¿Qué necesita un hospital para atender pacientes bajo la Ley Ricarte Soto?"
Procedural
4 questions · medium-hard
+0.64
What it tests: "What are the steps to…?" — ordered sequences that span multiple articles. RAG may return the right chunks but in the wrong order, losing the procedural flow.
Why GRAIL wins: Relationship chains encode process flow: Subsecretaría → evaluación → Comisión → recomendación → decreto. The agent uses local+cascade to capture both the process entity and the specific steps.
Example: "Si un hospital me hizo daño, ¿qué tengo que hacer antes de poder demandarlo?"
Comparative
3 questions · easy-hard
+0.80
What it tests: "What's the difference between X and Y?" where X and Y are defined in different places. A single RAG search finds one side of the comparison; the other side requires a second, well-targeted search.
Why GRAIL wins: The agent makes two local_search calls — one per concept — each crafted with the WHO+WHAT+TERMS formula. RAG searches once and hopes both sides land in the top-10 chunks.
Example: "¿Quiénes asesoran sobre garantías de salud y quiénes vigilan el sistema de alto costo? ¿Son los mismos?"
Negation / Boundary
3 questions · easy-medium
+0.08
What it tests: "Does the law cover X?" where the answer is NO. Tests whether the system retrieves exclusion clauses. Embedding similarity may miss negations because the query is semantically close to the covered concept.
Near-tied: Both systems handle explicit exclusions well. The exclusion clauses (Art. 1 "se excluyen") are prominent in the source text and match well with both entity descriptions and chunk similarity.
Example: "¿La Ley Ricarte Soto cubre los accidentes laborales o de tránsito?"
Global Synthesis
5 questions · easy-hard
+0.90
What it tests: Bird's-eye view of the entire corpus. "What is this about?", "What rights do I have?", "How do these 3 laws connect?" RAG has no single chunk that answers these — the answer IS the corpus.
Why GRAIL wins (largest delta): Global search reads pre-computed community report summaries that cover the entire knowledge base thematically. It answers "¿de qué trata?" in 6 seconds. RAG must piece together an overview from random top-10 chunks.
Example: "Haz un resumen de cómo estas tres normativas protegen a un paciente de cáncer."

Replicate This Benchmark

Run with the included dataset or bring your own corpus

With the included dataset

# 1. Index the oncology laws corpus
grail index benchmark_laws

# 2. Run the benchmark (agents only)
python benchmarks/run_benchmark.py \
  --config benchmark_laws/grail.yaml \
  --benchmark benchmarks/simple_benchmark/benchmark.json \
  --skip-rag --include-agents --skip-judge

# 3. Results saved to benchmarks/results/<timestamp>/

With your own corpus

# 1. Create and index your project
grail init my_project
cp my_documents/* my_project/input/
grail index my_project

# 2. Write a benchmark.json with your questions
#    (see benchmarks/simple_benchmark/ for format)

# 3. Run
python benchmarks/run_benchmark.py \
  --config my_project/grail.yaml \
  --benchmark my_benchmark.json \
  --skip-rag --include-agents --skip-judge

Judging Options

LLM-as-Judge (automated)
python benchmarks/run_benchmark.py \
  --config benchmark_laws/grail.yaml \
  --benchmark benchmarks/simple_benchmark/benchmark.json \
  --include-agents \
  --judge-model "openai|gpt-4o"

Any OpenAI-compatible endpoint works. The judge scores each response on 5 dimensions against gold answers. Fast, cheap, reproducible.

Claude Code as Judge (manual, highest quality)
# 1. Run with --skip-judge
python benchmarks/run_benchmark.py \
  --config benchmark_laws/grail.yaml \
  --benchmark benchmarks/simple_benchmark/benchmark.json \
  --include-agents --skip-judge

# 2. Ask Claude Code: "judge the benchmark"
# Claude reads responses.json + benchmark.json,
# scores each response, writes judge_scores.json

Highest quality scoring — Claude has full context of the source documents and understands domain nuances. Used for this benchmark.
